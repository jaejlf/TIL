# [MySQL] InnoDB의 버퍼 풀과 언두 로그
## 버퍼 풀(Buffer Pool)
- 디스크 I/O를 최소화하기 위해, 데이터 파일이나 인덱스 정보를 `메모리에 캐시`해두는 공간이다.
- 쓰기 작업을 지연 시켜, 일괄 작업으로 처리할 수 있게 해주는 `버퍼 역할`도 한다. (여기저기의 데이터를 변경하는, 랜덤한 디스크 I/O 작업의 횟수를 줄일 수 있다.)
- `LRU(Least Recently Used) 알고리즘`을 사용해서, 오랫동안 사용되지 않은 순으로 버퍼 풀에서 제거한다. (cf. MRU(Most Recently Used) 알고리즘)

## 언두 로그(Undo Log)
- 트랜잭션과 격리 수준을 보장하기 위해 DML(INSERT, UPDATE, DELETE) 전에, `변경되기 전의 데이터`를 언두 로그 영역에 백업한다. 
     - `UPDATE` : 업데이트되기 전의 데이터를 언두 로그에 기록    
    -  `INSERT` : 새로운 레코드를 삭제하는 작업을 언두 로그에 기록 (변경되기 전의 데이터 = 해당 레코드가 없는 상태이므로)
    - `DELETE` : 기존의 레코드를 다시 삽입하는 작업을 언두 로그에 기록
    - `SELECT` : 언두 로그에 기록이 되는 작업은 없다. (트랜잭션 격리 수준에 따라 언두 로그의 데이터를 조회할 수 있는지 없는지 달라짐!)
- ex. 대량의 DELETE 작업을 수행할 때, 삭제되기 전의 데이터를 언두 로그에 저장한다. (즉, 삭제하는 만큼 언두 로그 공간을 사용하게 되며 디스크의 사용량이 늘어날 것!)
    - ⛔ 대량 변경 작업으로 언두 로그가 꽉 차게 된다면, 데이터 변경 작업을 기록할 공간이 없으므로 다른 DML 작업에도 영향이 있을 수 있다.  
    - 언두 로그는 언제 정리가 될까? (트랜잭션이 커밋 된다고 바로 삭제되지는 않는다고 하며, purge 기간 설정을 조정할 수 있다고 한다. _더 찾아보기.._.)
- cf. 언두 로그가 저장되는 공간을 `언두 테이블스페이스(Undo Tablespace)`라고 한다.
